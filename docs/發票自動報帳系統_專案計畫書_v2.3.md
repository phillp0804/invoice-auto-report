# 發票自動報帳系統 — 專案計畫書

版本:v2.3(員工上傳 / 總務彙整雙角色版,含n8n自動化流程、QR Code優先辨識、信心分數確認機制)
日期:2026年7月
專案類型:AI課程作品集專案

---

## 1. 專案背景與動機

### 1.1 問題描述
一般公司報帳流程常見痛點:
- 員工需自行整理發票、填寫報帳單,流程繁瑣
- 總務/會計需人工核對統編合法性、加總分類、檢查重複報帳,耗時且容易出錯
- 發票分散在不同員工手上,月底彙整困難

### 1.2 專案動機
本專案模擬真實企業報帳情境,設計「員工只需上傳、總務直接取得整理好的報表」的自動化流程,降低雙方的操作負擔,同時作為AI應用開發與自動化工程能力的作品集展示。

### 1.3 與「直接詢問AI」的差異(核心價值論證)
若僅將發票照片丟給大型語言模型詢問,可得到單次分析結果,但無法達成以下目標,而這些正是本系統的核心價值:

| 需求 | 單次問答AI | 本系統 |
|---|---|---|
| 多人上傳彙整成單一報表 | 無法跨使用者彙整 | 系統自動彙整全員工上傳資料 |
| 跨月份累積統計 | 無記憶 | 資料庫累積、依月份自動加總 |
| 統編合法性驗證 | 可能誤判 | 程式碼實作檢查碼演算法,結果確定 |
| 重複發票偵測(跨員工) | 無法比對歷史/他人資料 | 全公司範圍比對發票號碼 |
| 分類一致性 | 同問題可能不同答案 | 固定分類規則 + AI輔助模糊判斷 |
| 角色分權(員工/總務看到的東西不同) | 無角色概念 | 員工只管上傳,總務取得彙整結果 |
| 結構化報表輸出 | 純文字回覆 | 可依部門/員工/類別匯出 Excel |

## 2. 專案目標

### 2.1 主要目標
開發一套雙角色系統:
- 員工端:拍照上傳發票即完成,不需額外操作
- 總務端:直接取得系統自動辨識、驗證、分類、彙整完成的報表,僅需審核例外項目

### 2.2 次要目標
- 展示多角色權限設計與自動化流程整合能力
- 展示「AI應用邊界」的思考(哪些交給AI、哪些用確定性邏輯處理)
- 產出可作為求職作品集的完整Demo

### 2.3 本階段範圍
本計畫聚焦於拍照上傳路徑,財政部電子發票API串接列為未來擴充項目(P2)。

## 3. 角色與使用情境

### 3.1 角色定義

| 角色 | 權限與操作 |
|---|---|
| 員工 | 僅能拍照上傳發票、查看自己上傳的紀錄與狀態,無其他操作 |
| 總務 | 可查看全公司彙整報表、審核例外項目(驗證失敗/疑似重複)、匯出報表 |

### 3.2 使用者流程

員工端流程:
```
開啟App
  ↓
拍照或上傳發票
  ↓
系統偵測圖片是否有QR Code
  ├─ 有QR Code → 直接解碼(準確度最高)
  └─ 無QR Code → 走AI辨識(Claude Vision),並回傳各欄位信心分數
  ↓
信心足夠(或QR解碼成功)
  → 直接視為完成,顯示「上傳成功」,員工不需額外操作
信心不足(關鍵欄位辨識不確定,例如統編/金額模糊)
  → 跳出確認/修正畫面,標示信心不足的欄位,員工確認或手動修正該欄位後送出
  ↓
[可選] 查看自己歷史上傳紀錄與狀態(已入帳/待確認)
```

補充說明:確認/修正畫面**不是預設路徑**,只在AI辨識信心不足時才出現,多數清晰的發票(尤其電子發票有QR Code)會直接完成,不影響「員工只需上傳」的核心體驗。

總務端流程:
```
登入總務後台
  ↓
查看本月彙整儀表板
  - 總金額、依類別分組金額
  - 依員工/部門分組明細
  - 例外項目清單(統編驗證失敗、疑似重複發票)
  ↓
對例外項目逐一確認或退回(退回時需填寫原因,系統會透過n8n通知該員工重新上傳)
  ↓
一鍵匯出報表(Excel,依部門/類別分組)
```

## 5. 系統架構

### 5.1 架構圖

```
員工端 App (拍照上傳)  ──▶
                        後端 API (FastAPI)
                              │
                              ▼
                    QR Code 偵測(pyzbar)
                     ├─ 有QR → 直接解碼(高準確度)
                     └─ 無QR → Claude Vision辨識 + 信心分數
                              │
                              ▼(信心不足才觸發)
                    員工確認/修正畫面
                              │
                              ▼
總務端後台 (彙整/審核/匯出) ◀── 規則驗證引擎(統編/重複/年份)
                                       │
                                       ▼
                              資料庫(含角色/狀態欄位)
                                       │
                                       ▼ (狀態變化觸發 webhook)
                              ┌─────────────────┐
                              │      n8n         │
                              │ - 退回通知員工     │
                              │ - 定期彙整寄送     │
                              │ - 待審核提醒       │
                              └─────────────────┘
```

### 5.1.1 n8n 負責的自動化流程(核心系統之外的擴充層)

n8n 不參與辨識、驗證、分類這些核心邏輯(這些留在 FastAPI 後端,確保結果可控、可測試),只負責「事件觸發後的外圍自動化動作」,設計原則是核心系統用程式碼寫、周邊流程用n8n串:

| 流程 | 觸發方式 | n8n 執行動作 |
|---|---|---|
| 退回通知員工(F15) | FastAPI 在狀態變更為「已退回」時呼叫 n8n Webhook | n8n 發送 Email 或 Line Notify 給對應員工 |
| 定期彙整寄送(F13) | n8n 排程(每月1號觸發) | 呼叫後端報表API取得Excel,自動寄送至總務信箱 |
| 待審核數量提醒(F16) | n8n 排程(每日檢查) | 呼叫後端API查詢待審核件數,超過門檻時通知總務 |
| 未來:雲端發票同步(F14) | n8n 排程 | 定期呼叫財政部API,寫入後端資料庫 |

### 5.2 技術選型

| 模組 | 技術 | 說明 |
|---|---|---|
| 影像辨識 | Claude API(多模態) | 無QR Code時的備援辨識路徑,直接讀圖片辨識文字 |
| QR Code解碼 | pyzbar / zxing | 優先辨識電子發票QR Code,準確度最高,同時降低AI呼叫成本 |
| 後端框架 | FastAPI (Python) | 需加入角色權限中介層(middleware) |
| 規則引擎 | 純Python邏輯 | 統編檢查碼、民國年轉換、跨員工重複比對 |
| 身份驗證 | Firebase Authentication | 區分員工/總務角色(可用自訂欄位標記角色) |
| 資料庫 | SQLite(開發)→ PostgreSQL(正式) | 需設計 users、invoices、departments 等資料表 |
| 前端 | React (Web) | 員工端與總務端為兩組不同介面(可用路由權限區隔) |
| 報表匯出 | openpyxl / pandas | 產出Excel報表 |
| 自動化/通知/排程 | n8n | 負責退回通知、定期彙整寄送、待審核提醒等周邊流程,不參與核心辨識/驗證邏輯 |
| 部署 | Render / Railway | 免費層即可 |

| name | TEXT | 姓名 |
| email | TEXT | Email(對應Firebase帳號) |

| department | INTEGER (FK) | 對應 departments.id |

| 欄位 | 型別 | 說明 |

| invoice_number | TEXT | 發票號碼(用於重複偵測) |
| tax_id | TEXT | 統編 |
| tax_id_valid | BOOLEAN | 檢查碼驗證結果 |
| date | DATE | 發票日期(已轉西元年) |

| category | TEXT | AI判斷的支出類別 |

| 欄位 | 型別 | 說明 |

| id | INTEGER (PK) | 主鍵 |

## 7. 關鍵技術細節

### 7.1 角色權限設計
使用 Firebase Authentication 登入後,依 users 資料表中的 role 欄位決定前端顯示的介面與後端API的存取權限(例如總務專屬的彙整/匯出API需驗證 role 為 admin)。

### 7.2 跨員工重複發票偵測
重複偵測範圍從「單一使用者」擴大為「全公司」,以 invoice_number 作為唯一鍵查詢整個 invoices 資料表,不限定上傳者本人。

### 7.3 AI分類與Prompt設計原則
- 要求AI僅輸出固定JSON格式
- 提供預先定義的分類清單,限制AI在固定選項內判斷
- 對辨識信心不足的項目標示「無法確定」,交由總務審核而非強行分類

### 7.4 n8n Webhook 安全性
n8n呼叫後端webhook路由時,後端必須驗證請求標頭中的密鑰(對應 `.env` 中的 `N8N_WEBHOOK_SECRET`),避免此端點被偽造請求觸發異常通知或狀態變更。

### 7.5 圖片上傳前處理
手機拍照的發票影像通常解析度高、檔案大(可能達5-10MB),直接送給Claude Vision會增加Token消耗與延遲,甚至可能超過API限制。上傳後、送交AI辨識前,應先將圖片壓縮(限制最長邊約1500px並轉為JPEG),兼顧辨識準確度與成本控制。

### 7.6 QR Code優先辨識機制
台灣電子發票(收銀機列印)通常印有QR Code,內含結構化資料(發票號碼、日期、隨機碼、統編、金額等),準確度接近100%。系統應優先偵測並解碼QR Code,只有偵測不到QR Code(例如手寫發票、傳統三聯式發票、部分小型店家舊式收銀機)時,才轉交Claude Vision辨識。QR Code資料格式需依財政部電子發票QR Code編碼規範解析,開發時應查證最新規範文件。此設計除了提升準確度,也能降低AI呼叫次數與成本。

### 7.7 信心分數與Human-in-the-loop確認機制
無QR Code、走AI辨識路徑時,AI回傳結果需附上各欄位的信心分數(例如高/中/低)。確認/修正畫面**非預設路徑**,只有在關鍵欄位(統編、金額、發票號碼)信心不足時才觸發,員工僅需確認或修正被標示的欄位,不需要重新輸入整張發票。此設計讓系統在多數清晰發票情境下維持「員工只需上傳」的體驗,同時對模糊、手寫發票保留人工把關,避免AI誤判進入報表。

## 9. 風險與因應

| 風險項目 | 影響 | 因應方式 |
|---|---|---|
| 紙本/手寫發票辨識率不穩(無QR Code時) | 辨識錯誤影響報表正確性 | QR Code優先解碼降低多數風險;無QR時以信心分數觸發人工確認,不完全依賴自動辨識 |
| AI分類不一致 | 報表可信度降低 | 限制AI在固定分類清單內選擇 |
| 角色權限設計遺漏(員工看到他人資料) | 資安/隱私疑慮 | 後端API務必驗證角色權限,而非僅前端隱藏畫面 |
| 真實發票含個資/財務資料 | 展示Demo時的隱私疑慮 | 使用假資料或去識別化範例製作Demo |
| 財稅規則複雜度高 | 誤導使用者 | 明確標示系統僅做分類彙整,非稅務判斷建議 |

## 10. 預期成果與作品集呈現重點

1. 真實情境對應能力:對應實際企業報帳流程中的兩種角色需求,而非單一使用者的簡化情境
2. 權限與角色設計:展示不只是CRUD,還包含角色分權的系統設計思維
3. AI應用邊界的思考:確定性邏輯(統編驗證、重複偵測)與AI模糊判斷(支出分類)的分工設計
4. 對「LLM wrapper」質疑的回應能力:能具體說明多人彙整、角色分權是單次問答無法達成的價值
5. 端到端資料流完整度:從辨識、驗證、分類、儲存到報表輸出的完整系統設計
6. 核心系統與自動化工具的分工判斷:展示「哪些邏輯自己寫程式碼確保可控性(辨識/驗證/分類)、哪些交給n8n處理標準化的通知與排程流程」,呼應自動化工程領域常見的架構決策思維
