## 1. 文件目的

本文件定義程式碼的資料夾結構、模組職責切分、命名規則與開發規範,確保:
- 用 AI 輔助開發(Claude Code)時,生成的程式碼結構一致、不會每次都不一樣
- 核心邏輯(辨識/驗證/分類)與周邊功能(路由/權限/通知)清楚分離,方便測試與維護

本文件不涉及「為什麼做這個專案」、功能規格與系統架構圖,該內容請參考《專案計畫書》。本文件只處理「程式碼怎麼組織」。

## 3. 模組職責說明(後端)

### 3.1 分層原則
採用簡化版分層架構,職責分工如下:

| 層級 | 職責 | 不該做的事 |
|---|---|---|
| routers | 接收 HTTP 請求、驗證輸入格式、呼叫對應 service、回傳結果 | 不寫商業邏輯,不直接操作資料庫 |
| services | 商業邏輯核心(辨識、驗證、分類、彙整) | 不處理 HTTP 相關細節(如 status code) |
| models | 定義資料表結構(ORM) | 不含邏輯運算 |
| utils | 純函式,無外部依賴(方便單元測試) | 不呼叫資料庫或外部API |

### 3.2 各 service 詳細說明

**qr_service.py**
- 職責:偵測上傳圖片中是否有QR Code,若有則呼叫 `utils/qr_parser.py` 解碼,回傳結構化資料(視為100%可信,不需信心分數)
- 若偵測不到QR Code,回傳「無QR Code」結果,交由 `ocr_service.py` 接手辨識

**ocr_service.py**
- 職責:當 `qr_service.py` 偵測不到QR Code時,接收圖片,呼叫 Claude API 多模態辨識,回傳結構化資料(發票號碼、日期、統編、金額、品項)
- 前處理:呼叫圖片前,先將圖片壓縮(限制最長邊約1500px並轉為JPEG),降低Token消耗與避免超過API限制
- 輸出格式:固定 JSON schema,對應 `schemas/invoice_schema.py`,並附上各欄位信心分數(高/中/低),供前端判斷是否需要人工確認

**validator_service.py**
- 職責:呼叫 `utils/tax_id_checksum.py` 驗證統編、呼叫 `utils/date_converter.py` 轉換日期、查詢資料庫比對是否有重複發票號碼
- 這是確定性邏輯,不使用 AI 判斷,方便寫單元測試保證正確性

**classifier_service.py**
- 職責:呼叫 Claude API,依照固定分類清單判斷支出類別
- 內含 Prompt 模板,固定要求輸出 JSON,並限制在預先定義的類別選項內

**report_service.py**
- 職責:依部門/員工/類別彙整資料,產出 Excel 檔案(使用 openpyxl 或 pandas)

## 4. API 路由規劃

| 路由 | 方法 | 說明 | 權限 |
|---|---|---|---|
| /auth/login | POST | 登入(搭配 Firebase token 驗證) | 公開 |
| /invoices/upload | POST | 員工上傳發票圖片(自動走QR或AI辨識) | 員工 |
| /invoices/{id}/confirm | POST | 員工確認/修正信心不足的欄位(僅信心不足時觸發) | 員工 |
| /invoices/my | GET | 員工查看自己的上傳紀錄 | 員工 |
| /admin/dashboard | GET | 總務彙整儀表板資料 | 總務 |
| /admin/invoices/pending | GET | 待審核例外項目清單 | 總務 |
| /admin/invoices/{id}/approve | POST | 總務確認項目 | 總務 |
| /admin/invoices/{id}/reject | POST | 總務退回項目(需帶 `reject_reason` 欄位) | 總務 |
| /admin/reports/export | GET | 匯出 Excel 報表 | 總務 |
| /webhook/n8n/invoice-status | POST | 供 n8n 訂閱狀態變化(需驗證 `X-Secret-Token` 標頭) | 系統內部 |

## 5. 命名規則

- 檔案/資料夾:全部小寫,單字間用底線(snake_case),例如 `invoice_router.py`
- Python 類別名稱:大駝峰(PascalCase),例如 `class InvoiceService`
- Python 函式/變數:小寫底線(snake_case),例如 `def validate_tax_id()`
- React 元件檔案:大駝峰(PascalCase),例如 `UploadPage.jsx`
- 資料庫欄位:小寫底線,例如 `created_at`、`tax_id_valid`
- API路由路徑:全小寫,用連字號(kebab-case),例如 `/admin/invoices/pending`

## 7. 開發規範建議

1. **先寫 utils 再寫 service**:像 `tax_id_checksum.py` 這種純函式邏輯,先寫好並搭配單元測試,確保正確性後再串進 service 層
2. **service 層要能獨立測試**:不要讓 service 直接依賴 FastAPI 的 `Request` 物件,方便之後寫 pytest
3. **AI 呼叫集中管理**:`ocr_service.py` 和 `classifier_service.py` 都呼叫 Claude API,建議統一寫一個 `services/claude_client.py` 共用底層呼叫邏輯(API Key讀取、錯誤處理、重試機制),避免重複程式碼
4. **Prompt 獨立管理**:建議把 Prompt 模板放在獨立的 `services/prompts/` 資料夾內(例如 `prompts/classify_invoice_prompt.py`),方便之後調整不用改動邏輯程式碼

## 9. 技術選型決定

- ORM:採用 SQLAlchemy(搭配 SQLite 開發、PostgreSQL 正式環境),與 `models/` 資料夾的分層設計一致
- 前端狀態管理:採用 React 內建 Context API 處理登入狀態與角色權限,MVP規模不需引入 Redux/Zustand
- n8n workflow JSON 匯出檔納入版本控制,存放於 `n8n-workflows/` 資料夾,證明專案包含完整自動化系統設計,不只是後端CRUD
- AI 廠商採可切換設計:`claude_client.py`/`gemini_client.py` 實作相同介面(`send_text`/`send_image`),`ai_client.py` 依 `AI_PROVIDER` 設定挑選,`ocr_service.py`/`classifier_service.py` 只依賴共用介面,不寫死特定廠商。原因是 Claude API 無永久免費額度,Gemini 有免費額度,課程作品集場景需要可以低成本測試,同時保留 Claude 作為預設選項
